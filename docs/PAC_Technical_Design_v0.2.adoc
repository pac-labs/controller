= PAC Technical Design Document
Version: 0.3

== Purpose

This document defines the architecture, terminology, and execution model of the PAC platform.
It is intended as a single source of truth for developers and agents working on the system.

== Platform Definition

PAC is a local-first, distributed agent orchestration platform.

It enables:
- Execution of tasks across multiple endpoints
- Structured use of tools within controlled workspaces
- Integration with external model providers
- Iterative planning and execution through agents
- PAC-backed development sessions from IDEs and the Web UI
- Customer- and user-scoped working contexts

PAC does NOT:
- Host or run language models itself
- Replace endpoint capabilities
- Treat the controller as a general-purpose compute node for arbitrary heavy work

== Architecture Overview

Components:
- Controller
- Agents
- Endpoints
- Endpoint Containers
- Workspaces
- Tools
- Plugins
- Model Providers (external)
- IDE Integrations

The Controller coordinates. Agents decide and execute. Endpoints perform work.

== Core Components

=== Controller

Central orchestration layer.

Responsibilities:
- Task planning
- Agent coordination
- Endpoint selection
- State management
- Session bootstrap for IDE/Web initiated work
- Lightweight operational preparation for execution environments

Constraints:
- No model hosting or inference
- No broad parallel heavy execution as a default controller role

Allowed controller-local work:
- Configuration management
- Session orchestration
- State persistence
- Artifact indexing
- Small control-plane actions such as bootstrap, wrapper management, metadata collection, and targeted container launch requests

Non-goals for the controller:
- Running large user workloads directly by default
- Acting as the main endpoint for all code execution
- Becoming the primary location for long-running heavy agent fleets

=== Agents

Agents are the core intelligence layer.

Types:
- Controller Agent (planning)
- Endpoint Agent (execution)
- Subagents (scoped workers attached to a session or endpoint/container execution context)

Responsibilities:
- Interpret tasks
- Select tools and endpoints
- Interact with model providers
- Iterate based on results
- Delegate bounded work to subagents when appropriate

Clarification:
- PAC may coordinate multiple external model providers for a single task lifecycle.
- Intelligence is externalized to providers, but PAC remains responsible for orchestration, routing, control, and safety decisions.

=== Endpoints

Execution environments.

Characteristics:
- OS-agnostic (Linux, Windows, etc.)
- Tool-capable
- Workspace-hosting
- Self-contained execution

Endpoints must:
- Register with controller
- Expose available tools
- Allow secure command execution
- Be addressable through the controller so PAC can route session work to them

Endpoints may:
- Be physical or virtual machines
- Be Windows, Linux, or other supported operating systems
- Support local tool execution only
- Support container execution
- Host endpoint-local agents or wrappers that maintain connectivity to the controller

Endpoint containers:
- Are valid execution environments for PAC sessions
- May be launched on a selected endpoint for isolated work
- May host a scoped worker/subagent runtime for the lifetime of a session or stage

=== Workspaces

Scoped execution environments.

Properties:
- Contain context (code, docs, data)
- Assigned to endpoints
- Used by agents for execution
- Can be:
  - Persistent
  - Ephemeral
- May be customer-scoped and user-scoped

Purpose:
- Provide bounded context for tasks
- Carry the right customer documentation, repository state, and user-specific context for the selected session

=== Tools

Executable binaries available on endpoints.

Rules:
- Installed locally
- Versioned
- Discoverable by PAC

Grouped into:
- Tool packages (dev, ops, etc.)

=== Plugins

Extend agent capabilities.

Examples:
- Parsing logic
- Data processors
- Workflow helpers

Plugins are:
- Not binaries
- Loaded by agents
- Contextual

=== Model Providers

External systems providing AI/model capabilities.

Examples:
- Remote APIs
- Local inference servers

PAC usage:
- Agents send requests
- Agents interpret results

PAC does not manage models directly.

Provider expectations:
- PAC can route different stages or subtasks to different model providers.
- PAC should collect lightweight provider and endpoint metadata so operators can understand hardware fit, health, and whether a chosen provider is under stress.

== Execution Model

1. Task received
2. Controller Agent creates execution plan
3. Plan split into stages
4. For each stage:
   - Select endpoint
   - Select workspace
   - Select tools
   - Optionally select model provider
   - Optionally select or launch an endpoint container
   - Optionally attach or create a scoped subagent
5. Endpoint Agent executes tools
6. Results returned to Controller Agent
7. If needed:
   - Send data to model provider
   - Receive interpretation
8. Iterate until completion

Typical PAC-backed IDE flow:

1. User opens PAC integration in Zed or VS Code
2. User selects a session type, such as `docs/customer_name`
3. User selects a profile, such as `doc-reader`
4. PAC resolves:
   - customer context
   - user context
   - target repository/workspace
   - execution endpoint
   - model/provider profile
5. PAC starts or reuses an endpoint container when isolation is required
6. PAC materializes the repository and context into that execution environment
7. A scoped endpoint-side worker or subagent is attached to the session
8. Session state and progress are visible in the PAC online system and IDE

== Communication Model (Draft)

- Controller <-> Endpoint Agent:
  - Secure API (HTTP/gRPC)
  - Task dispatch + result return

- Agent <-> Model Provider:
  - API-based interaction

- Endpoint:
  - Executes locally
  - Reports back

- IDE <-> Controller:
  - Session initiation
  - Session attachment
  - Timeline/status streaming
  - Workspace/profile selection

== Design Principles

- Local-first execution
- Explicit execution boundaries
- Agent-driven logic
- Externalized intelligence (models)
- Modular extensibility
- Secure communication
- Visible orchestration across IDE, controller, endpoint, and container boundaries
- Customer- and user-aware session bootstrapping
- Guided creation flows: every UI mechanism that adds or creates a resource starts from a compact `+` action and opens a guided modal instead of exposing large inline forms
- User profile settings: personal profile / My Access must be a coherent settings workspace with identity context, section navigation, and focused panels, not a stack of loosely connected cards.

== Constraints

- No internal model hosting
- Endpoint must support required tools
- Workspace defines scope
- Network required for model access (unless local provider exists)
- PAC should prefer endpoint-local execution over controller-local execution for user workloads
- User-facing sessions should be reproducible from explicit profile, workspace, endpoint, and context selection

== Known Gaps (Next Iteration)

- Workspace lifecycle definition
- Security model (auth, secrets, isolation)
- Agent capability schema
- Tool discovery standard
- Container-backed IDE session lifecycle
- Customer/user context materialization contract
- Provider and endpoint metrics model

== Notes for Future Agents

- Do not assume models run inside PAC
- Always respect workspace boundaries
- Prefer endpoint-local execution
- Treat tools as authoritative execution layer
- Keep controller control-plane oriented, but do not confuse "lightweight" with "unable to perform operational bootstrap work"

=== Windows Endpoint Execution

Windows endpoints are first-class PAC endpoints for workloads that can run on Windows. They register as endpoint principals, advertise the `windows` workload platform, and execute host jobs through the controller-queued endpoint job channel. PowerShell is the default host shell for Windows endpoints.

Rules:
- Windows remote code execution must remain endpoint-scoped and permission-gated through Directory & Access.
- Tokens and certificates authenticate the endpoint principal only; directory grants decide what the endpoint can do.
- Jobs should run in the configured endpoint workspace by default.
- Endpoint capabilities must advertise OS family, remote code execution availability, default shell, and available tools.

=== Endpoint Inventory UI Rule

The endpoint inventory page must show endpoints as workload cards, not as old runner rows or placeholder pills.

Each endpoint card should make current operational state visible:

- platform/OS family
- reachability and command-channel state
- default workspace
- pi.dev/runtime readiness
- container runtime availability
- tool inventory state
- hardware summary
- linked models/workspaces/directory principal

Status indicators should describe a concrete state and consequence. Avoid generic placeholder-looking pills such as plain `enabled`, `missing`, or unlabelled tags when a richer state row or capability tile can explain what the value means.

Verbose raw runtime metadata belongs behind a technical details disclosure so the inventory card stays usable for normal platform operators.

== Endpoint Operations and Observability UI Rule

Endpoint actions must not silently queue remote work without feedback.

Any endpoint action that starts or queues work should show a progress surface containing:
- endpoint identity
- job id
- operation name
- current status
- command or managed operation details
- recent progress events
- output/error when available

PAC uses local rotating Python logs as the default internal observability substrate. Logs are stored under `PACP_HOME/logs` and can be tailed through diagnostics APIs or shipped to an external system later.

=== Observe Navigation Ownership

Runtime evidence belongs under the existing Observe menu.

Observe includes:
- event rail and notifications
- controller-local logs
- metrics snapshots
- model usage
- traces and endpoint job evidence as those stores mature

Config remains for configuration and setup values. Do not create a second observability surface under Config/System when the data is operational evidence.

=== Embedded Observe Store Rule

PAC should provide a lightweight internal observability substrate before requiring an external observability stack.

The default internal layer is:
- rotating controller and audit logs under `PACP_HOME/logs`
- embedded SQLite metrics and trace spans under `PACP_HOME/observability.db`
- explicit retention and pruning controls
- APIs and UI under the existing Observe menu

This internal layer should remain dependency-free and low overhead. External systems such as VictoriaMetrics, VictoriaLogs, VictoriaTraces, Loki, Prometheus, or OpenTelemetry should be optional exporters/integrations, not required runtime dependencies for a normal PAC installation.


=== Observe Navigation

Observe contains runtime evidence: events, logs, metrics, traces, and diagnostics. Config contains configuration and setup values.

Events shown from the top notification/bell rail must include backend events and UI-triggered operations that are persisted through the controller event API. UI actions that say “Details are in Events” must create an event row, not only a transient browser-only message.

== Binary and Workspace Agent Model

PAC exposes two installable runtime binaries:

- `pac-endpoint`: the endpoint/workspace wrapper that runs where work happens.
- `pacctl`: the client used by humans, scripts, IDEs, containers, and automation to talk to PAC.

The former `pac-endpoint-runner` binary is removed. Its behavior is embedded in `pac-endpoint`.

`pac-agent` and `zed-binary` source folders have been retired. Their active behavior now belongs to the two primary binaries: `pac-endpoint` for endpoint/workspace execution and `pacctl` for CLI/MCP/client integration.

=== pac-endpoint

`pac-endpoint` supports host and workspace modes.

Host daemon mode registers the machine as an endpoint, forwards heartbeat, reports tools and capabilities, executes controller-routed jobs, and will forward hardware/metrics inventory.

Workspace mode uses:

----
pac-endpoint workspace run
----

This command is the foreground process for workspace containers. It reads workspace configuration from JSON or environment variables, registers the workspace with PAC, opens an outbound controller connection, sends heartbeat, executes PAC-routed commands, and keeps the container alive. When the container stops or loses connectivity, PAC marks the workspace offline after heartbeat timeout.

Workspace containers do not own durable command history. Command history, output events, session state, audit records, and PAC RAM live in PAC. Workspace storage should be centralized or mounted.

=== pacctl

`pacctl` communicates with PAC and asks PAC to route work to endpoints or workspaces. It should not bypass PAC by directly connecting to hosts by default.

Supported client responsibilities include:

- polling PAC state and events;
- sending provider/model information;
- using dynamic API/catalog endpoints;
- requesting endpoint/workspace commands through PAC;
- serving editor/MCP integration behavior after the Zed bridge is merged.

=== Release delivery

GitHub Releases publish direct binary assets for `pac-endpoint` and `pacctl` per OS/architecture, plus `RELEASE_BINARIES.json`. Source/update zips intentionally do not bundle compiled binaries. Installation and update flows resolve matching binary assets from GitHub and only build from source as an explicit fallback.

=== Endpoint telemetry and workspace liveness

`pac-endpoint` owns lightweight host and workspace telemetry. In host daemon mode it forwards hardware inventory, tool readiness, and runtime metrics with endpoint heartbeats. In `workspace run` mode it registers the workspace, keeps the outbound command loop open, and forwards the same telemetry shape with workspace-agent heartbeats.

PAC stores command history, PAC RAM, events, and workspace liveness centrally. Workspace containers are ephemeral executors: when the container stops or the outbound tunnel breaks, PAC marks the workspace degraded/offline from heartbeat age instead of relying on local state in the container.


== Workspace command live streaming

Workspace command execution is routed through PAC. `pacctl` queues commands against a workspace and may either poll for completion or open a live command stream. The workspace-side `pac-endpoint workspace run` process claims queued commands, executes them inside the workspace root, forwards ordered stdout/stderr events to PAC, and then posts the final command status and exit code.

PAC stores command events, aggregated output, status, and exit code centrally. Workspace containers are allowed to be ephemeral: if the container exits, heartbeat timeout marks the workspace offline, while command history and PAC RAM remain in the controller.

=== Workspace live terminal interruption

Workspace commands are routed through PAC and can now be streamed and interrupted. The controller keeps the command history and ordered event stream, while the workspace agent polls PAC for cancellation intent during execution. A queued command can be marked `interrupted` immediately; a claimed command receives `cancel_requested` metadata and the endpoint terminates the local process, completing the command with an interrupted status and exit code 130.

Endpoint inventory and metrics from `pac-endpoint` are surfaced as endpoint detail cards in the UI. The command event model still needs a dedicated high-volume event table for long-running or noisy commands; this follow-up is documented in `docs/workspace-command-event-model-followup-20260529.md`.

=== Workspace command event store

Workspace command output is stored in PAC as append-only command events keyed by workspace id, command id, and sequence number. The live terminal and `pacctl workspace exec --stream` replay from this store. Command records keep bounded recent events and aggregate stdout/stderr summaries only, which keeps command history available after workspace containers stop without growing command payloads indefinitely.

== Update Orchestrator and Binary-First Release Flow

PAC release updates are source-first and binary-resolved. The source/update zip must not bundle compiled binaries. GitHub Actions builds the first-class release binaries before packaging and publishes them as direct release assets plus `RELEASE_BINARIES.json`.

First-class binaries:

* `pac-endpoint` - endpoint/workspace wrapper.
* `pacctl` - PAC client, integration, provider, workspace, and MCP tool.

Default release targets:

* `linux/amd64`
* `linux/arm64`
* `darwin/amd64`
* `darwin/arm64`
* `windows/amd64`

When the Update Center applies a release, PAC also runs an environment orchestration phase. It downloads `RELEASE_BINARIES.json`, resolves and verifies the matching `pac-endpoint` and `pacctl` assets for the controller host platform, installs them into the local PAC binary cache, regenerates PAC/pi.dev tool instructions, and verifies the pi.dev runtime state. This keeps pi.dev/tool guidance aligned with the installed PAC environment without requiring bundled binaries.
